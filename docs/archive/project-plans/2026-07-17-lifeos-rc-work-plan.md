# LifeOS RC 候选版工作计划与原子任务清单

日期：2026-07-17  
来源：[项目进展评估与后续处理方案](../../project-review/2026-07-17-project-progress-review.md)  
计划状态：已归档，待执行  
目标版本：LifeOS v1.0 RC 候选版

## 一、范围与默认假设

本计划基于当前代码库和已完成的项目评估制定。当前没有必须先向用户澄清的阻塞项，因此采用以下默认口径：

- 本轮目标是内部可验收的 RC 候选版，不直接承诺正式公开发布。
- 优先保障本地数据安全、数据可恢复、核心流程稳定和打包应用可验证。
- 优先支持 macOS 主平台；Windows / Ubuntu 在 RC 后进行兼容性补测。
- 云同步、AI、移动端、插件市场、网页剪藏等列入后续路线图，不进入本轮发布门禁。
- 现有单元测试、构建和 lint 继续作为每个阶段的基础门禁。

## 二、总体目标

完成以下四个结果后，项目才可进入内部试用：

1. 密码保险箱不再明文保存，旧数据有明确迁移策略。
2. 用户数据具备可操作的备份、恢复和升级迁移方案。
3. 正式更新链路、安装包和版本发布流程完成真实演练。
4. 核心用户旅程在打包应用中通过，且无 P0 缺陷、无数据丢失和阻塞性崩溃。

## 三、阶段计划与退出条件

| 阶段 | 名称 | 目标 | 退出条件 | 依赖 |
|---|---|---|---|---|
| A | 安全与数据保护 | 消除高风险明文存储并补齐恢复边界 | 安全检查通过，迁移测试通过 | 无 |
| B | 规格收敛 | 让 v1.0 承诺与实际能力一致 | 规格矩阵完成，mock 已处理 | A 并行可开始 |
| C | 发布工程 | 完成构建、签名、更新和回滚演练 | 两版本升级演练通过 | A |
| D | 打包应用验收 | 验证真实桌面用户旅程 | P0 场景全通过 | A、B、C |
| E | RC 内部试用 | 收集真实反馈并决定是否发布 | 试用问题完成分级和决策 | D |

建议执行顺序：A 与 B 并行，A 完成后推进 C，B / C 完成后进入 D，D 完成后进入 E。

## 四、角色定义

- 产品经理（PM）：范围、优先级、验收标准、规格矩阵、发布决策。
- 项目经理（TPM / PM）：排期、依赖、风险、任务状态、变更控制。
- Electron / 安全开发：加密、IPC、数据目录、备份恢复、更新链路。
- 前端开发：设置入口、状态反馈、迁移提示、搜索和体验收口。
- QA：测试用例、异常场景、打包验收、缺陷分级和回归。
- 发布负责人：签名、产物、更新源、版本说明和内部分发。

## 五、原子任务清单

### 阶段 A：安全与数据保护

#### A-01 盘点保险箱现有数据格式

- 负责人：Electron / 安全开发
- 优先级：P0
- 输入：`src/views/Toolbox.tsx`、`electron/db/schema.ts`
- 动作：确认 vault 表字段、现存明文数据、调用链和迁移入口。
- 交付物：数据格式说明、风险清单、迁移策略草案。
- 验收：列出所有敏感字段、读写位置和旧数据处理方式。

#### A-02 选择并记录加密方案

- 负责人：Electron / 安全开发
- 优先级：P0
- 动作：在 AES-256-GCM 与 Electron `safeStorage` 之间完成选型，说明密钥生命周期、跨平台差异和失败行为。
- 交付物：安全设计记录。
- 验收：方案能回答“密钥存在哪里、主密码是否落盘、无法解密时怎么办”。

#### A-03 实现真实加密与解密服务

- 负责人：Electron / 安全开发
- 优先级：P0
- 依赖：A-02
- 动作：新增独立加密模块，提供 encrypt / decrypt，禁止 UI 直接写入敏感字段。
- 交付物：加密服务代码、类型定义。
- 验收：明文不进入数据库；错误密钥不能解密；篡改密文能被识别。

#### A-04 替换保险箱解锁逻辑

- 负责人：前端开发 + Electron / 安全开发
- 优先级：P0
- 依赖：A-03
- 动作：移除固定 `admin` 和“长度达到 4 位即可解锁”的逻辑，接入真实验证和锁定状态。
- 交付物：保险箱 UI 与 IPC 调整。
- 验收：正确密码可解锁；错误密码失败；重复错误有节流或锁定；退出或超时后重新锁定。

#### A-05 设计旧数据迁移

- 负责人：Electron / 安全开发
- 优先级：P0
- 依赖：A-01、A-03
- 动作：识别旧版明文记录，启动时提示用户重新保护或重新录入；不得伪装成已加密。
- 交付物：迁移脚本 / 迁移流程、用户提示文案。
- 验收：旧数据库升级不崩溃，迁移可重复执行，迁移失败可回滚。

#### A-06 补充敏感数据安全测试

- 负责人：QA + Electron / 安全开发
- 优先级：P0
- 依赖：A-03、A-04、A-05
- 动作：覆盖空密码、错误密码、密文篡改、迁移失败、剪贴板自动清理和锁定恢复。
- 交付物：自动化测试与人工检查表。
- 验收：测试全部通过，数据库抽查无明文密码。

#### A-07 定义用户数据边界

- 负责人：产品经理 + Electron / 安全开发
- 优先级：P0
- 动作：明确数据库、书籍、视频、配置、缓存、日志的目录和是否纳入备份。
- 交付物：数据目录与备份边界说明。
- 验收：用户能够理解哪些数据会被备份、恢复和删除。

#### A-08 实现最小备份

- 负责人：Electron / 安全开发
- 优先级：P0
- 依赖：A-07
- 动作：提供选择目标目录、生成备份包、写入版本与校验清单、展示进度和失败原因。
- 交付物：备份 IPC、设置入口、备份文件格式。
- 验收：任务、笔记、书籍记录、视频记录和必要配置可被完整导出。

#### A-09 实现最小恢复

- 负责人：Electron / 安全开发 + 前端开发
- 优先级：P0
- 依赖：A-08
- 动作：提供备份选择、版本兼容检查、恢复前确认、失败回滚和恢复完成提示。
- 交付物：恢复 IPC、UI 流程、恢复日志。
- 验收：新用户目录可恢复；恢复后应用重启可正常读取；失败不会破坏原数据。

## 七、项目管理规则

### 任务状态

- 未开始：已定义但尚未投入。
- 进行中：已有负责人且正在处理。
- 待验收：实现已完成，等待 QA 或产品验收。
- 已完成：验收证据已归档。
- 阻塞：依赖外部权限、证书、真实服务或明确的产品决策。

### 完成定义（Definition of Done）

一个原子任务只有同时满足以下条件才可标记为完成：

- 代码或文档已提交到对应位置。
- 有自动化测试或人工验收证据。
- 中英文文案已补齐，若适用则完成无障碍检查。
- 失败路径和回滚行为已验证。
- 相关风险、已知限制和后续事项已记录。

### 变更控制

- 新增功能必须说明对 RC 范围、排期和风险的影响。
- P0 安全或数据问题可以打断当前排期。
- P2 需求不得挤占 P0 / P1 发布门禁。
- 任何“已完成”状态必须附文件、测试或验收记录。
