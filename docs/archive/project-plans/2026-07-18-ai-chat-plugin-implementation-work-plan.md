# LifeOS AI 对话插件实施工作计划

日期：2026-07-18

状态：已批准执行

需求基线：`docs/archive/product/2026-07-18-ai-chat-plugin-product-design-spec.md`

任务清单：`docs/archive/project-plans/2026-07-18-ai-chat-plugin-atomic-tasks.md`

目标分支：`codex/ai-chat-plugin`

## 一、执行目标

在现有 LifeOS Electron 应用的工具箱中交付一个可用、可配置、可恢复的 AI 对话插件，支持：

- 模型供应商列表与配置管理。
- Agent 配置与运行。
- MCP HTTP/SSE/stdio 配置、工具发现和调用。
- 多轮流式文本对话。
- 工具调用授权、取消、超时和诊断。
- 图片与视频生成任务在对话中的展示、本地缓存和恢复。
- 会话历史、媒体存储、备份、清理、国际化和无障碍。

## 二、执行原则

### 2.1 原子任务原则

- 每个任务只解决一个清晰问题。
- 每个任务必须能独立测试和回滚。
- 每个任务完成后单独提交，不将多个未验证任务压入同一提交。
- 后续任务只能建立在已通过质量门禁的前置任务上。
- 若发现规格冲突，先更新归档文档和任务清单，再继续实现。

### 2.2 测试驱动原则

- 数据模型、状态机、验证器、安全边界和适配器优先编写测试。
- UI 任务至少提供可测试的纯函数、状态转换或静态回归测试。
- 外部供应商使用本地 mock server 验证协议；真实 API 验收在用户提供凭据后补充。
- MCP 使用可控的测试服务覆盖成功、超时、拒绝和崩溃路径。

### 2.3 提交原则

每个任务提交前必须满足：

1. 针对性测试通过。
2. `npm test` 通过。
3. `npm run lint` 通过。
4. `npm run build` 通过。
5. `git diff --check` 通过。
6. 工作区只包含该任务相关变更。

如果全量测试因明确的环境条件无法执行，必须记录原因并至少完成可执行的等价验证；不得把未知失败作为通过。

### 2.4 现有功能保护

每个阶段结束后至少回归：

- 应用启动、登录和用户切换。
- 工具箱番茄钟、换算器和密码保险箱。
- 任务、笔记、书库和视频页面切换。
- 视频本地播放和下载状态逻辑。
- 备份/恢复数据边界。
- 中英文和四套主题。
- 800×600 最小窗口与覆盖层行为。

## 三、架构落地边界

### 3.1 Renderer

只负责页面与状态展示：

- 供应商、Agent、MCP 配置界面。
- 会话列表、消息时间线和输入区。
- 流式增量合并。
- 工具授权对话框。
- 图片查看和视频播放。

Renderer 不得：

- 读取完整 API Key。
- 直接请求模型供应商。
- 创建 stdio 进程。
- 下载带认证的媒体。
- 接收任意 shell 或任意 SQL 能力。

### 3.2 Preload/IPC

只暴露窄化、结构化和可校验的 AI API：

- 配置 CRUD 与连接测试。
- 会话 CRUD。
- 启动、停止、重试运行。
- 工具授权响应。
- 运行、工具和媒体事件订阅。
- 媒体打开、另存和定位。

### 3.3 Electron 主进程

负责：

- AI 数据库和迁移。
- Electron `safeStorage` 凭据服务。
- 供应商适配器。
- Agent 运行器。
- MCP 连接和进程生命周期。
- 工具权限策略。
- 媒体下载、转码、封面和资源协议。
- 中断任务恢复和诊断。

## 四、阶段安排

### 阶段 A：数据与配置基础

任务：AT-01 至 AT-07。

交付：

- AI 领域类型、运行时校验和错误模型。
- 独立 `ai.db` schema。
- 凭据加密服务。
- 供应商、Agent 和 MCP 配置服务。
- 配置 IPC 桥接。

退出条件：

- 配置数据按用户隔离。
- 密钥不以明文写入数据库。
- 供应商依赖关系和默认能力约束生效。
- 配置服务全量测试、lint 和 build 通过。

### 阶段 B：配置界面与文本会话

任务：AT-08 至 AT-14。

交付：

- AI 工具箱入口和独立工作区。
- 供应商列表及配置编辑。
- Agent 和 MCP 配置编辑。
- 会话与消息持久化。
- OpenAI-compatible 流式适配器。
- Agent 文本运行器和停止机制。
- 日常聊天 UI。

退出条件：

- 使用 mock provider 连续完成 10 轮对话。
- 流式消息、停止、重试和会话切换正确。
- 应用重启后历史可恢复。
- 工具箱其他三个插件无回归。

### 阶段 C：MCP 与权限

任务：AT-15 至 AT-17。

交付：

- Streamable HTTP/SSE/stdio MCP 连接。
- 工具发现、缓存和状态诊断。
- Agent 工具调用循环。
- 高风险工具授权 UI。

退出条件：

- HTTP 和 stdio 测试 MCP 均可发现并调用工具。
- 只读自动、危险确认、拒绝、超时和进程退出路径通过。
- 达到工具循环上限时安全停止。

### 阶段 D：图片与视频

任务：AT-18 至 AT-21。

交付：

- 本地 AI 媒体目录与安全资源协议。
- 图片生成、下载、缓存和画廊。
- 视频任务轮询、下载、探测、转码和播放。
- 重启恢复与媒体状态管理。

退出条件：

- Base64、远程 URL 和结构化媒体结果通过。
- 临时 URL 失效后本地资源仍可使用。
- 视频任务支持完成、取消、失败重试和重启恢复。
- 资源协议无法访问用户 AI 目录以外文件。

### 阶段 E：产品化与最终验收

任务：AT-22 至 AT-25。

交付：

- 媒体容量统计、清理与备份边界。
- 首次引导、响应式和动效。
- 国际化、键盘和无障碍。
- 安全、集成、跨模块和打包验收记录。

退出条件：

- 需求基线中的首版必须项均有实现证据。
- 全量测试、lint、build 和 Windows 包配置检查通过。
- macOS 关键用户旅程完成。
- 不存在 P0/P1 数据安全、崩溃或现有模块阻塞问题。

## 五、质量门禁

### 5.1 单任务门禁

- 新增行为有测试。
- 错误路径不只依赖手工观察。
- 变更不暴露密钥或任意执行能力。
- IPC 输入在主进程执行运行时校验。
- 数据库迁移对已有用户数据是增量和幂等的。

### 5.2 阶段门禁

- 阶段内全部原子任务已提交。
- 完成阶段级冒烟测试。
- 检查提交边界和未跟踪文件。
- 更新任务清单状态和已知风险。

### 5.3 最终门禁

- `npm test`。
- `npm run lint`。
- `npm run build`。
- `npm run verify:native`。
- `npm run verify:package:win`。
- 打包应用关键路径人工验证。
- 凭据、SSRF、本地协议、路径穿越和 Markdown 注入专项检查。

## 六、依赖与技术选型

### 6.1 必需依赖

- `@modelcontextprotocol/sdk`：MCP 客户端和标准传输。
- `gsap` 与 `@gsap/react`：仅用于首次引导和受控状态转场。
- Electron 内置 `safeStorage`：供应商和 MCP 凭据加密。
- Node 24 内置 fetch/stream：模型与媒体网络请求。
- 现有 better-sqlite3、marked、DOMPurify、FFmpeg 和视频协议基础。

### 6.2 依赖约束

- 新依赖必须锁定在 package-lock。
- 不为单一小功能引入大型 Agent 框架。
- 不让供应商 SDK 渗透通用领域层；优先使用协议适配器。
- 任何原生依赖必须通过 macOS 与 Windows 构建检查。

## 七、外部服务验证策略

### 7.1 自动化验证

- 使用本地 HTTP server 模拟 OpenAI-compatible 流。
- 使用测试 MCP server 模拟工具发现、成功、超时和崩溃。
- 使用本地 fixture 模拟图片 Base64、签名 URL 和异步视频任务。

### 7.2 真实服务验证

用户提供凭据后执行：

- 一个 OpenAI-compatible 文本模型。
- 一个 xAI/Grok 或等价图片生成接口。
- 一个视频生成接口。
- 一个真实远程 MCP 和一个真实 stdio MCP。

没有真实凭据时，不阻塞协议实现和自动化验收，但最终交付说明必须清楚区分“mock 端到端通过”和“真实供应商已验证”。

## 八、风险管理

| 风险 | 预防与处理 |
| --- | --- |
| 任务范围过大 | 原子任务独立提交，阶段门禁后再进入下一阶段 |
| 主进程变得臃肿 | AI 服务分目录，不继续堆叠到 `electron/main.ts` |
| Toolbox 单文件继续膨胀 | AI 页面独立组件和样式，Toolbox 只保留入口 |
| MCP 任意命令风险 | 配置与执行分离，模型不能修改命令，高风险逐次授权 |
| API Key 泄露 | safeStorage、Renderer 隔离、日志脱敏和专项测试 |
| 媒体占用过大 | 临时文件、磁盘预检、容量上限和清理 |
| 外部协议变化 | 适配器与 fixture 测试，错误映射不依赖页面 |
| 跨平台 stdio 差异 | 命令数组化、路径测试、Windows 构建门禁 |

## 九、进度维护

- 原子任务状态以任务清单为准。
- 每完成一项，将状态从“待开始”更新为“已完成”，记录提交哈希和验证命令。
- 若任务拆分发生变化，保留原编号并说明替代关系，不静默重编号。
- 每个阶段结束后更新阶段结论和遗留风险。

## 十、完成定义

AI 对话插件只有在以下条件同时满足时才视为完成：

1. 首版必须功能全部实现。
2. 供应商列表、Agent 和 MCP 可以由用户管理。
3. 文本、图片和视频消息均有端到端验证。
4. MCP 工具权限闭环有效。
5. 应用重启、用户切换和异常恢复行为明确。
6. 现有 LifeOS 模块回归通过。
7. 全量质量门禁通过。
8. 所有原子任务均有独立提交和验证记录。
