## 一、结论摘要

LifeOS 已完成从 UI 原型到可运行 Electron 应用的主要跨模块集成，当前处于“功能集成完成、体验与发布质量收口”的阶段。

建议当前版本暂定为：**内部验收候选版，不建议直接作为正式发布版**。

综合判断：

- 工程实现度：约 80%。主框架、数据层、主要模块和关键交互已经形成。
- 产品闭环度：约 65%—70%。核心模块可用，但部分规格仍停留在演示、模拟或未闭环状态。
- 发布准备度：约 55%—60%。构建链路已通，但安全、真实环境验收、升级和数据保护仍需补齐。

当前最重要的工作不是继续扩展模块，而是完成安全整改、规格差异收敛、真实桌面场景验收和发布准备。

## 四、当前主要问题与风险

### P0：必须在正式发布前解决

#### P0-1 密码保险箱仍为模拟加密，存在明文保存风险

证据位置：`src/views/Toolbox.tsx`。当前 `password_encrypted` 字段直接写入用户密码，`iv` 和 `tag` 只是随机字符串；解锁逻辑还接受固定的 `admin` 或任意长度达到 4 位的密码。

处理建议：

1. 使用 Node `crypto` 的 AES-256-GCM 或 Electron `safeStorage` 实现真实加密。
2. 主密码只用于派生密钥，不得保存明文。
3. 迁移旧数据时明确标记为不可信并要求用户重新录入，不能静默假装已加密。
4. 增加“错误次数、锁定、自动锁定、剪贴板清理、升级迁移”测试。
5. 在正式发布前完成一次数据库抽查，确认不存在明文密码。

#### P0-2 正式升级链路尚未被真实验证

开发环境中的检查、下载、安装更新是 mock 流程；生产环境虽然调用 `electron-updater`，但仓库内没有看到完成签名、发布源、版本升级和失败回滚的验收记录。

处理建议：

- 明确发布平台和更新源，配置签名证书、发布权限和版本策略。
- 用两个真实版本完成升级演练：旧版本安装、检查更新、下载、重启安装、数据保留、失败恢复。
- 开发环境的 mock 必须在 UI 上明确为开发模拟，避免被误判为正式能力。

#### P0-3 备份 / 恢复与数据迁移没有形成可交付闭环

产品定位强调本地数据所有权，但当前缺少清晰可操作的备份、恢复、迁移、版本升级和损坏恢复方案。

处理建议：

- 定义用户数据目录、数据库、书籍文件、视频文件、设置文件的边界。
- 提供手动备份和恢复入口，至少支持导出到指定目录。
- 设计数据库 schema 版本、迁移失败回滚和恢复前确认。
- 对大文件采用清单 / 增量策略，避免备份过程阻塞界面。

### P1：应在下一轮收口

#### P1-1 产品规格与实际实现存在差异

设计规格中仍写有部分未实现或未闭环能力：

- Dashboard 卡片拖拽重排与用户自定义固定小工具未形成明确持久化闭环。
- 任务的自然语言解析当前仍标注为 mock / 简化实现，不应按完整 NLP 能力宣传。
- 任务中的部分模板和调度日志是演示数据，需要区分真实数据与示例数据。
- 全局搜索当前查询任务、笔记、书籍，但未覆盖视频；与“跨模块搜索”定位不完全一致。
- 规格中提到的多种笔记格式、无限层级子任务、回收站 30 天、跨笔记双向链接等能力，尚未全部在当前代码中得到同等强度的验证。

处理建议：

- 建立“规格—实现—验收标准”矩阵，逐项标记已完成、部分完成、未开始和延期。
- 对暂不做的能力从 v1.0 规格中移出，避免产品承诺过大。
- 对已实现但为演示的内容统一加上开发标记或改为真实数据来源。

#### P1-2 缺少端到端和打包后人工验收

现有测试以单元、逻辑、数据库和服务 smoke test 为主，没有覆盖完整用户旅程，例如：

- 首次启动 → 创建账户 → 登录 → 创建任务 → 重启后数据保留。
- 导入 EPUB / PDF → 打开阅读器 → 跳章 → 阅读进度 → 批注导出。
- 解析视频 → 下载 → 中断 / 失败 → 重试 → 本地播放。
- 中英文切换、四种主题、窄窗口和高 DPI 显示。
- macOS 安装包启动、权限、升级、卸载和数据保留。

处理建议：建立最小 P0 E2E 清单，先完成 macOS 主平台，再补 Windows / Ubuntu 差异验证。

#### P1-3 缺少持续集成与发布门禁

当前可见仓库中没有明确的 CI / release workflow 证据。建议至少加入：

- PR 必须通过 test、build、lint。
- 主分支构建产物检查。
- 依赖和 Electron 安全审计。
- macOS 构建 / 签名 / 产物校验流程。
- 发布版本变更日志和回滚说明。

#### P1-4 部分实现仍存在明显 mock / hardcode，需要收口

已发现的典型项包括：

- 工具箱汇率使用固定费率。
- 密码保险箱使用模拟 IV / tag。
- 自动更新开发流程使用模拟进度。
- 任务自然语言输入和部分预览内容属于演示或简化逻辑。

处理方式：每个 mock 建立替换任务或明确标记为“演示能力”，不得在发布说明中描述为完整生产能力。

### P2：可放入后续版本

- 云同步、移动端、AI 助手、插件市场、网页剪藏、日历集成、笔记 Git 版本控制。
- 更完整的跨模块标签、归档、回收站和统一搜索。
- 更复杂的拖拽编排和个性化 Dashboard。
- Windows / Ubuntu 平台的专属安装与行为优化。

这些内容不应阻塞当前 v1.0 的内部验收，但应进入产品路线图并明确不纳入本轮发布。

## 五、建议的下一阶段方案

### 阶段 A：发布阻塞项整改

目标：让敏感数据和发布链路达到可验收状态。

工作项：

- 完成密码保险箱真实加密与旧数据迁移策略。
- 完成备份 / 恢复最小闭环。
- 明确生产更新源、签名和升级演练。
- 建立发布配置检查表和敏感数据检查脚本。

完成标准：安全审查通过；旧版本到新版本升级成功；数据可恢复；不存在明文密码。

### 阶段 B：规格差异收敛

目标：把产品承诺和真实能力对齐。

工作项：

- 完成规格—实现—验收矩阵。
- 将全局搜索补齐视频，或调整产品描述。
- 清理 / 标记 mock 数据和简化能力。
- 对 Dashboard、任务提醒、笔记组织和归档能力做取舍。

完成标准：每项 v1.0 能力都有实现证据、验收用例和明确负责人。

### 阶段 C：真实场景验收

目标：验证打包应用而不是只验证源码。

工作项：

- 建立 macOS P0 用户旅程测试。
- 覆盖无网络、权限不足、工具缺失、文件损坏、下载失败等异常场景。
- 进行窄窗口、高 DPI、中文 / 英文、主题切换和键盘操作检查。
- 记录缺陷、严重级别、复现步骤和验收截图。

完成标准：P0 场景全部通过；P1 缺陷有明确延期决定；无数据丢失和阻塞性崩溃。

### 阶段 D：内部发布候选版

目标：形成可供少量用户试用的 RC 版本。

工作项：

- 更新版本号、变更日志、已知问题和安装说明。
- 生成并校验安装包。
- 完成一轮内部试用，收集使用频率、失败路径和用户反馈。
- 根据反馈决定正式发布或回退到继续修整。

## 六、建议的优先级与决策

建议批准以下顺序：

1. 先处理 P0-1 密码保险箱安全问题。
2. 同步设计备份 / 恢复最小方案。
3. 完成生产升级链路的真实演练。
4. 再做规格差异清单和跨模块搜索补齐。
5. 最后进行打包应用 E2E 验收和 RC 发布。

建议暂缓：云同步、AI、移动端、插件市场等扩展性工作，直到本地数据安全和 v1.0 闭环稳定。
